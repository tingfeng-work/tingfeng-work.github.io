# 滑动窗口：从“枚举子数组”到“利用单调性缩小搜索空间”

在系统复习双指针专题时，我发现：

> 相向双指针解决的是“两端逼近”的问题，而滑动窗口（同向双指针）解决的则是“连续区间”的问题。

一开始，我对刷滑动窗口题目的理解还停留在：

- 一个 left
- 一个 right
- 不断移动窗口

但随着完成：

- 长度最小的子数组
- 无重复字符的最长子串
- 乘积小于 K 的子数组
- 最小覆盖子串

等经典题后，我逐渐意识到：

> 滑动窗口的本质，并不是“维护窗口”，而是利用区间性质与单调性，动态缩小搜索空间。

这篇文章，我希望从：

```
为什么窗口可以移动？
为什么不会漏解？
什么时候扩展？
什么时候收缩？
```

几个角度，系统总结我对滑动窗口的理解。

------

# 一、为什么滑动窗口通常用于“子数组 / 子串”？

滑动窗口有一个非常明显的特点：

```
窗口内就是一个连续区间
```

因此它天然适合处理：

- 子数组
- 子串
- 连续区间

例如：

| 题目                | 本质       |
| ------------------- | ---------- |
| 长度最小的子数组    | 连续子数组 |
| 无重复字符最长子串  | 连续子串   |
| 最小覆盖子串        | 连续子串   |
| 乘积小于 K 的子数组 | 连续子数组 |

因为：

> 窗口移动时，只需要增删边界元素，就能维护整个区间状态。

这也是滑动窗口相比暴力枚举最大的优势。

------

# 二、滑动窗口的本质：动态维护一个“合法区间”

滑动窗口的核心思想可以总结为：

```
right 负责扩展窗口
left 负责收缩窗口
```

窗口中的元素始终满足某种条件。

典型模板：

```
int left = 0;

for (int right = 0; right < n; right++) {

    // 1. 新元素进入窗口
    update(nums[right]);

    // 2. 当窗口合法性转变
    while (窗口合法/不合法) {
        remove(nums[left]);
        left++;
    }

    // 3. 更新答案可以在 while 内、前或后
}
```

这里其实体现了：

> 滑动窗口的本质并不是“枚举”，而是“维护”。

------

# 三、209. 长度最小的子数组：第一次真正理解“单调性”

经典题目：长度最小的子数组

题目要求：

```
找到和 ≥ target 的最短连续子数组
```

------

## 为什么能使用滑动窗口？

这道题最关键的条件：

```
数组中的元素全为正数
```

这意味着：窗口越大，区间和越大

即：存在单调性

因此：

- right 扩展窗口 → sum 增大
- left 收缩窗口 → sum 减小

这就形成了：

```
合法 与 不合法 的状态转换
```

------

## 为什么答案出现在“收缩过程”中？

例如：

```
sum >= target
```

说明当前窗口已经合法。

但题目要求：

```
最短
```

因此需要：

```
尽可能缩小窗口
```

于是：

```
while (sum >= target) {
    ans = Math.min(ans, right - left + 1);

    sum -= nums[left];
    left++;
}
```

这里我第一次真正意识到：

> 滑动窗口的答案，很多时候恰恰出现在“状态变化的边界”。

因为：

- 当前合法
- 再缩一步可能就不合法

那么：当前窗口很可能就是极值答案

------

# 四、滑动窗口中的三类经典问题

在刷完整个专题后，我发现大部分滑动窗口题目都可以归纳为三类。

------

# 第一类：越长越合法

典型题目：长度最小的子数组

特点：

```
窗口越大，越容易满足条件
```

例如：

- 和越来越大
- 覆盖字符越来越全

因此：right 扩展使窗口更合法

而：left 收缩使窗口更不合法

------

# 第二类：越短越合法

典型题目：乘积小于 K 的子数组

由于：乘积会随着窗口扩大而增大

因此：窗口越大，越不容易满足条件

当：product >= k

时：必须不断收缩 left，直到重新合法。

------

## 一个非常重要的结论

在 713 中我第一次理解：

> 当窗口合法时，以 right 结尾的所有子数组都合法。

例如：

```
[left, right]
```

合法。

那么：

```
[left+1, right]
[left+2, right]
...
[right, right]
```

也全部合法。

因此：

```
ans += right - left + 1;
```

这里本质上利用的是：单调性带来的“批量统计”。

这是滑动窗口中非常重要的思想。

------

# 第三类：维护特定条件

典型题目：

- 无重复字符的最长子串
- 每个字符最多出现两次
- 最多 K 个重复元素

这类题的特点：

```
窗口是否合法
取决于窗口内部状态
```

例如：

```
字符是否重复
```

因此：

```
需要额外数据结构维护窗口状态
```

通常：HashMap、HashSe、t计数数组（最快优先考虑）

------

# 五、无重复字符最长子串：窗口状态维护

经典题：无重复字符的最长子串

这里窗口维护的是：

```
字符出现次数
```

例如：

```
map.merge(c, 1, Integer::sum);
```

当：

```
某字符出现次数 > 1
```

窗口非法。

于是：

```
while (map.get(c) > 1) {
    map.merge(s.charAt(left), -1, Integer::sum);
    left++;
}
```

------

# 六、我对刷滑动窗口最大的理解变化

以前我对刷滑动窗口的理解是：

```
一个左指针和一个右指针的同方向上的移动
```

现在我更关注：

```
窗口中的“状态”是什么？
```

因为真正重要的不是：

- left++
- right++

而是：

```
窗口移动后，
什么状态发生了变化？
```

例如：

| 题目 | 窗口状态     |
| ---- | ------------ |
| 209  | 区间和       |
| 713  | 区间乘积     |
| 3    | 字符出现次数 |
| 76   | 字符覆盖情况 |
| 1004 | 0 的数量     |

我越来越觉得：

> 滑动窗口其实是在动态维护一个“区间状态机”。

------

# 七、最难的一类：76. 最小覆盖子串

这是我认为最能体现滑动窗口思想的一道题。

因为它同时涉及：

- 窗口扩展
- 状态维护
- 合法判断
- 最优解更新

------

## 核心思想

窗口需要满足：

```
覆盖 t 中所有字符
```

right 扩展：

```
让窗口逐渐合法
```

left 收缩：

```
寻找更短答案
```

本质仍然是：

```
在合法与不合法之间寻找边界
```

这也是滑动窗口最核心的思想。

------

# 八、滑动窗口与相向双指针的统一理解

刷完整个专题后，我发现：

> 滑动窗口与相向双指针，本质其实是同一种思想。

二者共同点：

```
利用单调性缩小搜索空间
```

区别只是：

| 类型       | 指针方向   |
| ---------- | ---------- |
| 相向双指针 | 两边向中间 |
| 滑动窗口   | 同方向移动 |

------

# 九、Java 中的一些工程细节

在这次复习过程中，我还进一步巩固了：

------

## 1. 方法引用

例如：

```
Integer::sum
```

它本质上是：

```
(a, b) -> a + b
```

的简化写法。

属于：

```
Lambda 表达式的进一步优化
```

------

## 2. map.merge()

例如：

```
map.merge(c, 1, Integer::sum);
```

作用：

```
如果 key 不存在：
    放入 value

如果存在：
    使用给定规则更新 value
```

例如：

```
map.merge(c, -1, Integer::sum);
```

可以优雅地完成：

```
计数减一
```

相比：

```
map.put(c, map.get(c) + 1);
```

代码更加简洁。
