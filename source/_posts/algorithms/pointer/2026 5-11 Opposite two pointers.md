# 从“两数之和”到“接雨水”，理解双指针背后的单调性

在刷 LeetCode 的过程中，我对“相向双指针（Two Pointers）”一类题目进行了集中训练。

一开始，我对双指针的理解停留在：

- 数组有序
- left 和 right 相向移动
- 两数之和和小了 left++
- 两数和大了 right--

但真正系统刷完“两数之和”、“三数之和”、“接雨水”等经典题后，我逐渐意识到：

> 相向双指针的核心并不是“双指针”本身，而是利用问题中的“单调性”，在一次移动中排除一整片不可能的解空间。

这篇文章，我希望从“为什么可以移动指针”这个角度，总结我对刷双指针专题的一些理解。

------

# 一、两数之和：相向指针的起点

经典题目：两数之和 II

由于数组有序：

```
nums[left] + nums[right]
```

如果：

```
sum < target
```

说明当前和太小。

因为数组递增：

- right 左边的数只会更小
- 移动 right 没意义

因此只能：

```
left++
```

反之：

```
sum > target
```

则需要：

```
right--
```

这里本质上利用的是：

> 有序数组带来的单调性。

一次指针移动，可以排除掉 O(n) 个无效状态，因为单调性，当前状态不满足，移动另一侧指针是无效的，就一次排除了 n 个无效状态，因此时间复杂度从：

```
O(n²)
```

降为：

```
O(n)
```

------

# 二、三数之和：双指针与剪枝

经典题目：三数之和

核心思路：

- 固定第一个数
- 剩余部分转化为“两数之和”

即：

```
a + b + c = 0
```

转化为：

```
b + c = -a
```

这里第一次让我真正意识到：

> 双指针不仅是一种技巧，更是一种“降维思想”。

------

## 去重问题

三数之和最关键的问题之一是：

```
结果不能重复
```

因此：

```
if (i > 0 && nums[i] == nums[i - 1]) continue;
```

以及：

```
while (left < right && nums[left] == nums[left - 1]) left++;
```

本质上都是：

```
跳过重复状态
```

------

## 剪枝优化

例如：

```
if (nums[i] + nums[i + 1] + nums[i + 2] > 0) break;
```

由于数组已经排序：

- 当前最小三数和已经大于 0
- 后续只会更大

因此：

```
可以直接结束循环
```

这里体现的是：

> 利用单调性进一步缩小搜索空间。

------

# 三、四数之和：Java 整形溢出问题

在完成“四数之和”时，我第一次真正踩到了 Java 整形溢出的坑。

例如：

```
long sum = a + b + c + d;
```

如果：

```
a,b,c,d 都是 int
```

那么表达式会先以 int 运算。

即：

```
int + int + int + int
```

可能在赋值给 long 之前已经溢出。

Java 中：

> 表达式类型由参与运算的操作数决定，而不是赋值变量决定。

因此正确写法应该是：

```
long sum = (long)a + b + c + d;
```

或者：

```
long sum = 0L + a + b + c + d;
```

这类问题本质上已经不只是算法问题，而是：

```
语言机制问题
```

------

# 四、接雨水与盛水容器：木桶效应中的单调性

经典题目：

- 1. 盛最多水的容器
- 1. 接雨水

这类题与“两数之和”最大的不同在于：

> 它们的单调性并不来自数组有序。

而来自：

```
木桶短板效应
```

------

## 盛最多水的容器

面积：

```
面积 = 宽度 × min(leftHeight, rightHeight)
```

容量取决于更短的一边。

假设：

```
leftHeight < rightHeight
```

如果移动 right：

- 宽度减少
- 短板不变

面积一定不会更优。

因此只能移动：

```
left++
```

这里的核心仍然是：

> 利用单调性排除无效解。

------

# 五、验证回文串：指针移动时机

经典题目：

- 1. 验证回文串

这道题让我开始真正关注：

```
指针到底什么时候移动？
```

题目要求：

- 忽略非字母数字字符
- 忽略大小写

因此：

- 有时只移动一个指针
- 有时两个同时移动

例如：

```
if (!Character.isLetterOrDigit(s.charAt(left))) {
    left++;
}
```

这里我还学习到了一个非常实用的 Java API：

```
Character.isLetterOrDigit(char c)
```

用于判断字符是否为：

- 字母
- 数字

相比手动 ASCII 判断：

```
c >= 'a' && c <= 'z'
```

代码更加简洁，也更符合工程开发习惯。

------

# 六、我对刷双指针的理解变化

以前刷题时，我更关注：

```
这题是什么模板？
```

现在我会优先思考：

```
为什么可以移动这个指针？
为什么不会漏解？
```

因为我越来越觉得：

> 算法真正重要的不是“记住模板”，而是理解算法为什么成立。

------

# 七、总结：相向双指针的本质

我现在对相向双指针的理解，可以总结为一句话：

> 相向双指针的核心不是“双指针”，而是“利用问题中的单调性，不断缩小搜索空间”。

这种单调性可能来自：

| 题目            | 单调性来源     |
| --------------- | -------------- |
| 两数之和        | 数组有序       |
| 三数 / 四数之和 | 排序 + 剪枝    |
| 盛水容器        | 木桶短板效应   |
| 接雨水          | 左右最大值约束 |
| 回文串          | 搜索空间收缩   |

------
